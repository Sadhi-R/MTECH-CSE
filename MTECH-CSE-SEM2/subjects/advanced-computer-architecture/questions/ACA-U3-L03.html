Question

Explain superscalar pipeline design.

DefinitionSuperscalar pipeline fetches/decodes/issues multiple instructions per cycle using dynamic scheduling and multiple functional units.

Visual Example

Introduction

Extends scalar pipeline with register renaming, reservation stations, and reorder buffer.

Why It Is Needed

Achieves ILP beyond simple pipelining.

Working Principle

Front-end fetches wide instruction window; dispatch to reservation stations; execute out-of-order; commit in-order.

Step-by-Step Explanation

  • Wide fetch/decode (4–8 instr/cycle).
  • Register renaming eliminates WAR/WAW.
  • Reservation stations hold operands.
  • Multiple FUs execute in parallel.
  • ROB commits in program order.

Architecture / Diagram

Fetch→Decode→Rename→Dispatch→[RS→FU]→ROB→Commit

Algorithm

Tomasulo-style: RS wait for operands → issue to free FU → broadcast result

Flowchart

Start -> Define -> Process -> Validate -> Conclude -> End

Real-Time Example

4-wide CPU: issues ADD, MUL, LOAD, BRANCH in one cycle if FUs free.

Advantages

  • High ILP exploitation
  • Hides latency via OoO
  • Multiple FUs utilized

Disadvantages

  • Complex hardware
  • Power hungry
  • Diminishing returns (ILP wall)

Applications

  • Out-of-order CPUs
  • High-performance servers
  • Embedded application processors

Exam Tips

Draw pipeline with rename + RS + ROB — key diagram.

Common Mistakes

  • Confusing in-order issue with in-order commit.

Difference Table

Superscalar=wide issue | Scalar pipeline=1 instr/cycle peak

Time Complexity

IPC > 1 possible; limited by dependencies

Space Complexity

ROB, RS, physical register file

Interview Questions

  • Role of reorder buffer?

Memory Tricks

Superscalar = wide fetch + OoO execute + in-order commit
